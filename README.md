## Запуск

Нужен только Docker.

```bash
docker compose up --build
```

Поднимает PostgreSQL, прогоняет миграции, заполняет базу тестовыми данными и стартует сервер на **http://localhost:3001**.

Swagger UI доступен по адресу **http://localhost:3001/docs**.

Ну и желательно, чтобы у вас на этом порту ничего не было запущено, конечно же

## Попробовать руками

После запуска база уже содержит тестовые данные: 10 пользователей с разными наборами каналов, таймзонами и тихими часами, 20 регионов с региональными политиками. Лучше это конечно делать через сваггер, там все видно

**1. Посмотреть список пользователей**

```bash
curl http://localhost:3001/users
```

В ответе у каждого пользователя видны его регион, политики и предпочтения — это поможет подобрать интересный кейс для проверки.

Как альтернатива — можно подключиться к базе напрямую:

```
host: localhost:5432
user: postgres
password: postgres
database: notifications
```

**2. Проверить, можно ли отправить уведомление**

```bash
curl -X POST http://localhost:3001/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "userId": 1,
    "channel": "email",
    "type": "marketing",
    "timestamp": "2026-06-01T14:00:00Z"
  }'
```

Подставьте `userId` нужного пользователя, `channel` и `type` из его `enabled`-листа — и получите `allow`. Уберите тип из `enabled`, попадите в тихие часы или выберите регион с политикой — получите `deny` с причинами.

## Устройство проекта

```
HTTP-запрос
    └── Controller        # валидация входа, HTTP-ответы
         └── Repository   # работа с БД, по одному файлу на сущность
              └── Prisma  # ORM → PostgreSQL
```

**Доменные типы** (`src/types/`) — основной словарь сервиса: `Channel`, `ChannelType`, `PolicyType`, `Decision`, `Reason`.

**Сущности:** `User` → принадлежит `Region` (у региона есть `Policies`), имеет одну запись `Preferences`.

**Preferences** хранят:
- `enabled` — список разрешённых комбинаций `PolicyType` (например, `email_system`, `push_marketing`)
- `timezone` — IANA-таймзона пользователя, используется при проверке тихих часов
- `quie_hours_start` / `quie_hours_end` — временные границы тихих часов

**`POST /evaluate`** — основной эндпоинт. Принимает `{ userId, channel, type, timestamp }`, последовательно выполняет три проверки:

1. **Предпочтения** — есть ли комбинация `channel_type` в списке `enabled` пользователя?
2. **Тихие часы** — попадает ли timestamp в тихие часы с учётом таймзоны? (только для `marketing`; системные уведомления проходят всегда)
3. **Региональная политика** — запрещает ли регион пользователя данную комбинацию `channel_type`?

Возвращает `{ decision: "allow" | "deny", reasons: string[] }` со всеми причинами отказа.

## Что добавить для продакшена

- **Переход на NestJS** — в проекте уже применяются паттерны, характерные для NestJS (контроллеры, репозитории, DI-образная структура); миграция упростила бы поддержку Swagger (декораторы вместо ручного `swagger.json`), тестирование и общую масштабируемость да и валидацию за счет ДТО
- **Типизация** - Перевел бы еще некоторые типы на энамы, тк при расширении сервиса и внедрения логики это будет куда удобней. Сейчас там просто юнион
- **Доменное разделение** — при росте требований текущая плоская структура станет узким местом; стоит выделить bounded contexts (например, `preferences`, `policies`, `evaluation`) с собственными слоями и границами
- **Аутентификация** — API-ключ или JWT; сейчас любой может изменить настройки любого пользователя
- **Пагинация** на `GET /users`
- **Нормальное логирование** — заменить `console.log` на структурированный логгер (например, `pino`) с уровнями и correlation ID запроса. Сечайс это просто по требованиям надо было
- **Метрики** — счётчики allow/deny по каналу и типу, p99-латентность на `/evaluate`
- **Интеграционные тесты** — текущие тесты мокают репозитории; тесты против реальной БД помогут поймать проблемы с миграциями
